## Executive Recommendation

Keep M0.5 as the lightweight closed loop now described by the design docs:

`Backend OTel SDK -> Collector -> ClickHouse otel_traces -> project VIEW -> traces API`.

The current `spec.md` is directionally close, but should be tightened around the revised package boundary: trace semantics and emission live in `@codecrush/otel-conventions` and `@codecrush/otel`; physical storage/query stays in `infra/` and `apps/backend/modules/traces`. The current `plan.md` already picked up many of these constraints, but still needs cleanup so it does not preserve stale contract exports, over-specify future agent/tool behavior as implementation, update authoritative design docs as part of dev, or couple the SDK to API DTOs.

Evidence: M0.5 is explicitly the minimal OTel SDK, Collector `clickhouseexporter`, raw `otel_traces`, VIEW, traces module, and hello-span acceptance row in the roadmap [docs/design/002-implementation-roadmap.md:70](../../../../docs/design/002-implementation-roadmap.md#L70)-[73](../../../../docs/design/002-implementation-roadmap.md#L73). The trace observability doc repeats the exact lightweight flow and says no trace-normalizer worker or independent `traces` / `observations` / `sessions` materialized write tables for M0.5 [docs/design/004-trace-observability.md:22](../../../../docs/design/004-trace-observability.md#L22)-[37](../../../../docs/design/004-trace-observability.md#L37). The architecture states the app only emits OTLP, the SDK does not know physical storage, and ClickHouse VIEW / Trace API stay in `infra/` and `traces` [docs/design/001-rag-platform-architecture.md:99](../../../../docs/design/001-rag-platform-architecture.md#L99)-[116](../../../../docs/design/001-rag-platform-architecture.md#L116). Code organization gives the same two-layer split, with `packages/otel-conventions` and `packages/otel` separated from `infra/collector`, ClickHouse views, and backend traces API [docs/design/003-code-organization.md:152](../../../../docs/design/003-code-organization.md#L152)-[168](../../../../docs/design/003-code-organization.md#L168).
